# Inventory Search — Design

## Context

The public `/inventory` listing page currently supports only one filter: `?make=X` (generated by the homepage's Brand Scroller links), matched case-insensitively against `vehicle.make`. This spec adds a full search block above the vehicle grid so buyers can filter by year range, make, model, color, body type, and price range. It also adds the two vehicle attributes (`color`, `bodyType`) needed to support color/type filtering, since neither exists in the data model yet.

This builds directly on the Vehicle Detail Page (Phase 1) work already merged — same Firestore path, same `getVehicles`/`vehicleSchema` conventions, same Add Vehicle modal.

## Data model changes

**`Vehicle`** (`src/types/vehicle.ts`) gains two fields, both with zod defaults so existing vehicle docs (which lack these fields) keep parsing without a migration — same pattern as `fuel`/`transmission`/`status`:

```ts
export type BodyType = "sedan" | "suv" | "hatchback" | "pickup" | "coupe" | "furgon";

// Vehicle interface gains:
color: string;
bodyType: BodyType;

// vehicleSchema gains:
color: z.string().default(""),
bodyType: z.enum(["sedan", "suv", "hatchback", "pickup", "coupe", "furgon"]).default("sedan"),
```

New `BODY_TYPE_LABELS: Record<BodyType, string>` constant (Sedán/SUV/Hatchback/Pickup/Coupé/Furgón), alongside the existing `FUEL_LABELS`/`TRANSMISSION_LABELS`/`VEHICLE_STATUS_LABELS`.

## Dashboard: Add Vehicle modal

Two new fields, matching existing conventions:
- `color` — text input (same pattern as `version`/`location`)
- `bodyType` — `<select>` with the 6 body-type options (same pattern as `fuel`/`transmission`/`status`)

`createVehicleAction` reads and validates both via the updated `vehicleSchema`.

## Search block on `/inventory`

A new `InventorySearchForm` Client Component, rendered above the vehicle grid on `src/app/(site)/inventory/page.tsx`:

- **Año** — min/max number inputs.
- **Marca** — `<select>`, options are the distinct `make` values across the *full* inventory (not the currently-filtered result set — so narrowing one filter never causes another dropdown's options to collapse).
- **Modelo** — `<select>`, dependent on the selected Marca: when a Marca is chosen, only that make's models are listed (computed client-side from the full vehicle list passed in as a prop); with no Marca selected, all models are listed.
- **Color** — `<select>`, distinct `color` values across the full inventory.
- **Tipo** — `<select>`, the 6 fixed `BodyType` options plus a "Todos" default.
- **Precio** — min/max number inputs (US$).
- **Buscar** submit button, plus a **Limpiar filtros** link that navigates to `/inventory` with no query params.

On submit, the form builds a query string and navigates (`router.push`) to `/inventory?year_min=&year_max=&make=&model=&color=&bodyType=&price_min=&price_max=` — only non-empty fields are included as params. This extends the existing `?make=` convention already used by the homepage's Brand Scroller links (unchanged, still works standalone or combined with the new filters).

`src/app/(site)/inventory/page.tsx` (Server Component) reads all these params and applies them as sequential filters over `getVehicles`'s result before rendering the grid — same approach as the existing `?make=` filter, just with more conditions. Year/price filters are inclusive-range comparisons; make/model/color are case-insensitive exact matches; bodyType is an exact enum match.

Because `InventorySearchForm` needs the full unfiltered vehicle list to build its dropdown options (not just the current filtered results), the Server Component fetches vehicles once and passes both the full list (to the form, for building dropdown option sets) and the filtered list (to the grid) — no duplicate Firestore read.

## Explicitly out of scope

- Free-text/fuzzy search (e.g. a single search box matching against multiple fields at once) — this is structured filter-by-field only.
- Persisting/saving searches, or filter history.
- Sorting controls (price ascending/descending, etc.) — filtering only.
- Editing `color`/`bodyType` on existing vehicles — still add-only, matching the rest of the Inventory feature's scope.
- Mobile-specific collapsed/accordion filter UI — the form uses the same responsive grid/stacking conventions as the rest of the site; a dedicated mobile filter drawer is not part of this phase.

## Verification approach

Matches this codebase's established convention — no automated test runner exists:

1. `npx tsc --noEmit` (plus `npx next typegen` where a route's `PageProps`/`searchParams` shape changes) after each task.
2. Throwaway `scripts/manual-verify-*.ts` round-trips against real Firestore for the schema change (confirms legacy vehicle docs still parse with the new defaults).
3. Manual browser walkthrough: add a vehicle with color/bodyType via the dashboard modal, confirm the search form's dropdowns include the new values, run combinations of filters (year range, make+model dependency, color, body type, price range) and confirm the grid narrows correctly, confirm "Limpiar filtros" resets to the full list, confirm the Brand Scroller's existing `?make=` links still work standalone.
